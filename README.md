## 概要
Unityで設計を突き詰めていたらできたDIコンテナのようなものです。
重いのでリフレクションは使わず、インターフェースを駆使して制作しました。

## 開発のきっかけ
学校の課題でのチーム開発の際、メンバー間で技術への理解度に差があったため、
コーディング規約の共有などのすり合わせが上手く行かず、密結合なコードになってしまっていました。
そこで、全てのメンバーが保守性の高い疎結合なコードを書くことができるよう、技術の力を使おうと思いました。

## 機能
IInstaller、IInstallable、IBindable、IInitializableという4つのインターフェースを使うことで、
Unity標準のAwakeとStartだけでは不安定な初期化順を制御しようとしました。

・IInstaller…初期化処理の全体を担うインターフェースです。
これを実装したInstallerBaseを継承したクラスをプレイヤーなどのルートオブジェクトにアタッチし、
SerializeFieldなどで登録した参照を、InstallAllメソッドでコンテナに追加します。

・IInstallable…あまり使いませんが、自分や自分が持つ参照をコンテナに登録する時に使うインターフェースです。

・IBindable…コンテナから参照を取得し、自分が持つフィールドやプロパティに紐付けます。

・IInitializable…Unity標準のStartに相当します。参照を一通り解決した後で動かしたい最初の処理を記述します。

## 課題
ヒエラルキーに最初から設置されているオブジェクトに対しては動作しましたが、
Awakeで動くUnityBootStrapperや、オブジェクト同士の親子関係に依存しすぎてしまったため、
プレハブから動的に生成したオブジェクトに対して上手く注入できませんでした。
また、ActivationProxyによって制御できたと思っていたOnEnableのタイミングも初期化時しか制御できていませんでした。

## 学んだこと
これを作ってから、既存のVContainerやZenjectといったフレームワークの存在を知りました。
先に自分で作ってみたことで、そのようなフレームワークのありがたさや、
それがどれほど深く考えて作られているか理解するきっかけになりました。
また、依存解決の仕組みを自分で考えて形にしたことで、
夢であるゲームエンジン開発の第一歩を踏み出すことができたと思います。
